# Памятка переноса (read me first)

Ты — агент, который продолжает работу над проектом генерации **сопроводительных писем**
для кандидата Дмитрия Кочетова (аналитик: Product / Data / BI / исследования). Прочти этот
файл целиком — он объясняет, что здесь собрано, и как себя вести.

## Что это за проект (одной фразой)
Генератор сопроводительных писем под вакансию + итеративно дорабатываемый набор правил:
пользователь даёт текст вакансии → ты выдаёшь 2 варианта письма; пользователь даёт замечания →
ты превращаешь их в правила и впредь соблюдаешь.

## Что прочитать и в каком порядке
1. `external-prompts/cover-letter-chatgpt.md` — ГОТОВЫЙ промпт для генерации письма
   (роль, источники фактов, запреты, структура, стиль, self-check). Это твой основной движок.
2. `external-prompts/building-rules.md` — почему промпт такой: дистиллят всех правок
   (C-001…C-032) и изменений ядра. Раздел 10 — самые свежие правила, приоритетнее старых.
3. Факты кандидата: резюме и задачи Jira (если приложены). Только отсюда берёшь факты.
   В оригинальном проекте есть `inputs/facts-digest.md` — выверенный дистиллят кейсов с
   цифрами; если он дан — это основной источник фактов.

Если работаешь в полном репозитории (не только эта папка), движок лежит в `prompt/`:
`core.md` (ядро, высший приоритет), `rules.md` (банк формулировок), `corrections.md`
(правки, применять только статус `active`). Приоритет при конфликте: **core > corrections > rules**.

## Главные «нельзя» (критично)
- Не выдумывай факты, цифры, инструменты, результаты. Любая цифра — из источников.
- В резюме ЕСТЬ запретное (стройка/ГНБ, маркетинг/GA/Метрика оттуда, учебные проекты
  Практикума) — это использовать НЕЛЬЗЯ. Полный список — в блоке `<forbidden>` промпта.
- Каждый кейс ОБЯЗАН быть релевантен вакансии; лучше меньше кейсов, чем нерелевантные/выдуманные.
- Стаж аналитика ~4,5 года, не «16 лет».

## Команды пользователя (что делать, когда он пишет…)
- **«Сгенерируй письмо»** — собери письмо по `cover-letter-chatgpt.md`, выдай 2 варианта.
- **«Замечание: …»** — пользователь говорит, что не так. Сформулируй ОДНО атомарное правило +
  скоуп (always или условие), проверь на противоречия с ядром и прошлыми правилами, добавь
  правило (C-NNN) и перегенерируй письмо. При конфликте с ядром — не меняй ядро молча, спроси.
- **«режим редакции» / «ок»** — пользователь правит письмо прямо в файле (в проекте —
  `outputs/letters_edit.md`); по «ок» сравни с эталоном, разбери каждую правку
  «было → стало → правило → скоуп», покажи и жди подтверждения перед записью.
- **«проверь на противоречия»** — прогони все активные правила между собой и против ядра.
- **«повысить C-NNN до core»** — перенеси правило в ядро, в правках пометь `merged-into-core`.
- **«humanizer on/off»** — переключатель «очеловечивания» (живой стиль). По умолчанию on;
  логика humanizer уже вшита в промпт (блоки `<живость_и_голос>` и `<стиль>`).
- **«git save» / «git load»** — сохранить/восстановить контекст между сессиями (журнал
  `MEMORY.md` + `ARCHITECTURE.md`; в проекте ещё commit+push).

## Что НЕ переносится автоматически (если ты не Claude Code)
Команды и правила — это договорённости из текста, а не встроенные функции. Чтобы они
заработали у тебя, тебе должны отдать этот файл и промпт, и ты следуешь им сам. Специфика
Claude Code (скилл humanizer, звуковой хук, MCP, авто-память, авто-чтение `CLAUDE.md`) не
переедет: humanizer бери из промпта; `CLAUDE.md` у Codex называется `AGENTS.md` — его надо
явно дать. Качество соблюдения тонких правил зависит от модели — будь внимателен к стилю.

## Принципы работы (коротко)
Думай до действия (при неоднозначности — спроси, не угадывай; противоречия выводи, не сглаживай).
Минимализм. Точечные изменения. Работай от критерия: «устроило пользователя, факты честные,
кейсы релевантны вакансии».
